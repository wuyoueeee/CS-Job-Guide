# 项目实战专题：把“做过项目”变成“能拿来面试”

项目不是简历上的技术词堆叠，而是面试官判断你工程能力的证据。一个能讲清楚的项目，至少要具备四件事：

1. 有明确业务场景，而不是“为了用技术而做技术”。
2. 有可解释的架构设计，而不是照着教程搭一遍。
3. 有真实难点和取舍，而不是只完成 CRUD。
4. 有可验证结果和追问准备，而不是写完就结束。

本专题按“项目选择 → 架构拆解 → 面试追问 → 简历表达”的方式组织，目标是帮你把项目变成可展示、可复盘、可面试的作品。

## 项目面试评分标准

| 维度 | 低分表现 | 高分表现 |
| --- | --- | --- |
| 真实性 | 只能说用了什么技术 | 能讲清自己负责的模块、代码细节和排查过程 |
| 复杂度 | 只有增删改查 | 有并发、性能、一致性、权限、安全、可观测等问题 |
| 取舍 | “网上都这么做” | 能解释为什么选这个方案，替代方案是什么 |
| 结果 | 没有验证 | 有压测、日志、指标、测试集、错误案例 |
| 表达 | 技术词堆叠 | 背景、方案、难点、结果、反思结构清楚 |
