# 项目深挖面试准备清单：把项目讲到面试官愿意继续问

技术面试里，项目深挖往往比八股更能区分候选人。八股答得好，说明你准备过；项目讲得深，说明你真的做过、想过、踩过坑。

很多同学项目不差，但面试时讲不出来。问题通常不在项目本身，而在准备方式：只记技术栈，不讲业务；只讲功能，不讲权衡；只讲结果，不讲验证。

## 一、项目深挖的核心目标

面试官深挖项目，通常在验证五件事。

| 面试官想验证 | 他可能怎么问 | 你要证明什么 |
| --- | --- | --- |
| 真实性 | 这个模块是你做的吗？ | 你清楚细节，不是挂名 |
| 复杂度 | 难点在哪里？ | 项目不是简单 CRUD |
| 判断力 | 为什么这样设计？ | 你有技术取舍意识 |
| 结果意识 | 效果怎么验证？ | 你不是做完就结束 |
| 成长性 | 如果重做会怎么改？ | 你能复盘和升级方案 |

如果你的项目回答一直停在“用了什么技术”，面试官很难判断你是不是只照着教程做了一遍。

## 二、项目介绍的 3 分钟结构

项目开场不要讲太散。建议用这个顺序：

```text
背景问题 → 系统方案 → 我的职责 → 一个难点 → 结果验证
```

示例：

```text
这个项目是一个面向校园二手交易的后端系统，主要解决发布、搜索、私信和订单流转问题。
整体上采用 Spring Boot + MySQL + Redis，核心模块包括商品、订单、消息和用户权限。
我主要负责订单模块和商品搜索优化。
项目里最值得展开的是订单状态一致性和搜索接口性能优化。
在 10 万条模拟商品数据下，我们通过索引优化和缓存热点分类，将搜索接口平均响应时间从 700ms 降到 220ms。
```

这段开场的好处是：既交代背景，又主动给面试官递出可追问的点。

## 三、准备 10 个必问问题

每个主项目都要准备下面 10 个问题。答不上来，就说明项目还没有复盘到位。

| 问题 | 准备重点 |
| --- | --- |
| 这个项目解决什么问题？ | 不要只说“练手”，要说使用场景 |
| 为什么选择这个技术栈？ | 讲约束、团队熟悉度、替代方案 |
| 你本人负责哪一块？ | 区分个人贡献和团队成果 |
| 项目最难的地方是什么？ | 讲具体问题，不要泛泛说“并发难” |
| 你怎么定位问题？ | 现象、日志、指标、假设、验证 |
| 有没有做过性能优化？ | 指标、瓶颈、方案、结果 |
| 有没有考虑异常场景？ | 超时、重试、幂等、降级、补偿 |
| 数据库表怎么设计？ | 主表、关联表、索引、状态字段 |
| 如果流量扩大 10 倍怎么办？ | 瓶颈分析、拆分、缓存、异步 |
| 如果重做一次会改什么？ | 反思和改进，不要说“没问题” |

## 四、把技术点讲成“问题解决”

面试官不缺技术名词，他想听你怎么用技术解决问题。

| 普通表达 | 更好的表达 |
| --- | --- |
| 使用 Redis 做缓存 | 商品详情页读多写少，为降低 MySQL 压力，对热点商品详情做 Redis 缓存，并设置随机过期时间降低雪崩风险 |
| 使用 MQ 异步处理 | 下单后通知、积分和日志不影响主链路，因此拆到消息队列异步处理，同时保留失败重试和补偿任务 |
| 使用索引优化查询 | 针对订单列表按 user_id + create_time 查询的场景，增加联合索引并调整排序字段，避免全表扫描 |
| 使用分布式锁 | 为避免重复提交导致库存扣减异常，对关键扣减链路增加幂等 token 和 Redis 锁，但同时设置超时避免死锁 |

公式很简单：

```text
场景约束 + 技术方案 + 风险处理 + 验证结果
```

## 五、准备一个“失败案例”

项目里只有成功故事，反而显得不真实。准备一个失败或踩坑案例，会让你更像真正做过项目的人。

可以选这些类型：

| 类型 | 示例 |
| --- | --- |
| 性能问题 | 接口一开始很慢，后来发现是 N+1 查询 |
| 一致性问题 | 订单创建成功但库存扣减失败 |
| 边界问题 | 重复点击导致重复提交 |
| 协作问题 | 前后端字段理解不一致 |
| 设计问题 | 一开始表设计不合理，后面迁移成本变高 |

回答结构：

```text
当时出现了什么现象？
我怎么定位？
最终原因是什么？
采取了什么修复方案？
后续如何避免同类问题？
```

## 六、项目深挖错题本

每次项目被问住，都记录到错题本里。

| 字段 | 示例 |
| --- | --- |
| 被问住的问题 | 你们怎么保证订单状态一致性？ |
| 我当时怎么答 | 只说用了事务，没讲异常补偿 |
| 面试官继续追问 | 如果库存扣减成功但订单写入失败怎么办？ |
| 暴露的问题 | 对分布式一致性和补偿机制理解不足 |
| 补救动作 | 补充本地事务、幂等、重试、补偿任务的方案 |
| 下次标准回答 | 先讲单库事务，再讲跨服务失败处理 |

项目深挖不是一次准备完就结束。每次面试都是一次压力测试，把被问住的问题补回项目稿里，项目会越讲越扎实。

## 七、危险信号

如果你的项目回答出现这些情况，要尽快调整。

| 危险信号 | 说明 |
| --- | --- |
| 只能说“我们做了” | 个人贡献不清楚 |
| 技术点都能列，但讲不出为什么 | 可能只是堆栈 |
| 没有任何指标 | 结果不可验证 |
| 被问异常场景就卡住 | 工程意识不足 |
| 所有问题都说“没遇到过” | 项目可信度下降 |
| 项目讲完面试官没有追问 | 可能没有亮点或表达太平 |

## 八、面试前 20 分钟速查表

```markdown
## 项目深挖速查

- 项目一句话背景：
- 我的三个具体贡献：
  1.
  2.
  3.
- 最值得讲的技术难点：
- 一个真实踩坑案例：
- 一个性能或质量指标：
- 三个可能追问：
  1.
  2.
  3.
- 如果重做我会改：
```

面试时不要试图把所有内容都讲完。先讲主线，再把最有价值的点留给追问。
