# MySQL 高级高频面试题

这篇文章面向已经掌握索引、事务和锁基础的同学。不要只背结论，建议结合执行计划、表结构和实际 SQL 验证理解。

## 1、什么是聚簇索引和二级索引？

在 InnoDB 中，聚簇索引的叶子节点保存完整行记录。主键索引通常是聚簇索引。二级索引的叶子节点保存索引列和主键值，查询其他列时可能需要根据主键再次查找聚簇索引，这个过程通常称为回表。

追问：什么情况下可以使用覆盖索引避免回表？

## 2、为什么常见索引结构是 B+ 树？

B+ 树具有较低树高，适合减少磁盘 IO；数据集中在叶子节点，叶子节点有序连接，适合范围查询。回答时应说明它与哈希索引、普通二叉搜索树的差异。

## 3、什么是最左匹配原则？

联合索引会按照索引列从左到右建立有序结构。查询条件能否有效利用索引，与联合索引列顺序、范围条件和查询方式有关。

```sql
CREATE INDEX idx_user_city_age ON user(city, age);

SELECT id FROM user WHERE city = '杭州' AND age >= 20;
```

不要机械背诵规则，建议使用 `EXPLAIN` 验证不同 SQL 的访问方式。

## 4、如何阅读执行计划？

重点关注：

| 字段 | 关注点 |
| --- | --- |
| `type` | 访问方式，是否出现全表扫描 |
| `possible_keys` | 可能使用的索引 |
| `key` | 实际使用的索引 |
| `rows` | 预计扫描行数 |
| `Extra` | 是否出现额外排序、临时表等信息 |

## 5、什么是 MVCC？

MVCC 即多版本并发控制。理解时建议围绕版本链、读取规则和隔离级别展开，并区分一致性读与当前读。

追问：快照读和当前读有什么区别？加锁查询属于哪一种？

## 6、如何分析死锁？

```mermaid
flowchart LR
    A[记录死锁信息] --> B[还原事务顺序]
    B --> C[识别锁范围]
    C --> D[统一访问顺序]
    D --> E[缩短事务并验证]
```

常见改进思路：

1. 统一访问资源顺序。
2. 缩短事务范围和持锁时间。
3. 为查询条件建立合适索引。
4. 对可重试业务设计有限次数重试。

## 7、如何优化慢查询？

1. 明确问题 SQL 和执行频率。
2. 使用执行计划分析扫描范围。
3. 检查索引、返回列、排序和分页方式。
4. 评估是否需要改写 SQL、拆分查询或调整数据模型。
5. 上线后观察指标，确认优化是否有效。

## 8、深分页为什么可能变慢？

偏移量很大时，数据库可能需要扫描并丢弃大量记录。可以结合业务场景考虑基于游标或上一页末尾值的翻页方式。

```sql
SELECT id, title
FROM article
WHERE id > 100000
ORDER BY id
LIMIT 20;
```

## 9、主从复制需要关注什么？

面试中建议从日志、复制过程、延迟和一致性四个方面说明。不要只回答“用于读写分离”，还要说明复制延迟可能给业务带来的影响。

## 10、分库分表是不是首选方案？

不是。分库分表会引入路由、跨库查询、事务、数据迁移和运维复杂度。应先确认瓶颈，再评估索引、SQL、缓存、读写分离和数据归档等手段。
