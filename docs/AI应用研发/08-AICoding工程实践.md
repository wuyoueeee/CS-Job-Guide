# AI Coding 工程实践：让模型提高效率，不替你承担责任

AI Coding 已经成为研发工作流的一部分。对校招生来说，真正有价值的不是展示“我用过某个工具”，而是说明：

- 你把什么任务交给模型。
- 你如何提供上下文。
- 你如何审查结果。
- 你如何验证代码。
- 你如何保护仓库和数据。

一句话原则：

> 模型可以帮你加速理解、生成和审查，但代码责任仍然属于提交者。

## 一、适合交给 AI 的任务

| 任务 | 适合程度 | 注意点 |
| --- | --- | --- |
| 解释陌生模块 | 高 | 要求引用文件和调用链 |
| 生成重复性代码 | 高 | 检查边界、命名和项目风格 |
| 补充测试样例 | 高 | 检查是否覆盖真实失败场景 |
| 排查报错 | 中高 | 提供日志、版本和最小复现 |
| 修改核心交易逻辑 | 谨慎 | 必须人工审查、测试和回滚方案 |
| 直接执行删除、发布、迁移 | 高风险 | 需要明确确认和权限边界 |

AI Coding 不是把整个仓库交给模型后等待成品。高质量使用方式更像一位工程师带着一位速度很快、但会犯错的协作者。

## 二、一个可复用的工作流

```mermaid
flowchart LR
    A["说明目标"] --> B["读取仓库约定"]
    B --> C["定位相关代码"]
    C --> D["提出小范围修改"]
    D --> E["审查 diff"]
    E --> F["运行测试"]
    F --> G["人工检查风险"]
    G --> H["提交"]
```

### 目标要具体

不够好的请求：

```text
优化一下这个项目。
```

更好的请求：

```text
职位详情接口在 Redis 删除失败时没有重试。
请先读取现有缓存封装和测试风格，再补充有限次数重试。
不要修改接口返回结构。增加覆盖删除失败的测试，并说明风险。
```

上下文越具体，模型越容易做出可审查的改动。

## 三、一定要看 diff

AI 生成的代码可能：

- 看起来合理，但调用了不存在的 API。
- 修复了主路径，漏掉异常路径。
- 破坏已有行为。
- 引入越权访问。
- 把敏感信息写入日志。
- 顺手修改无关文件。

因此，每次改动后至少检查：

```bash
git status --short
git diff
git diff --staged
```

审查问题：

1. 改动是否只覆盖目标范围？
2. 是否改变公共接口？
3. 是否新增依赖？
4. 是否包含密钥、配置或个人信息？
5. 测试是否验证了失败路径？
6. 是否容易回滚？

## 四、测试不是最后一步

让 AI 写代码时，最好同时要求测试。

一个有效的测试任务：

```text
为缓存删除失败后的重试补充测试：
1. 第一次失败、第二次成功。
2. 连续失败超过上限。
3. 不可重试错误直接返回。
4. 验证没有无限重试。
```

测试能迫使实现暴露边界，也能防止后续修改悄悄退化。

## 五、让 AI 帮你做代码审查

AI 可以作为额外一层审查：

```text
请审查当前 diff，优先寻找：
1. 行为回归。
2. 并发问题。
3. 越权和敏感信息泄露。
4. 缺失的异常处理。
5. 缺失测试。
只报告可以落到具体代码位置的问题。
```

但 AI 审查不是最终批准。GitHub 官方文档也提醒，AI 生成或建议的代码可能存在错误或安全问题，应该仔细审查并测试。

## 六、仓库与数据边界

使用 AI Coding 工具前，先确认：

- 仓库是否包含密钥、生产配置或客户数据。
- 组织是否允许把代码发送给第三方服务。
- 工具会保存哪些上下文。
- 哪些目录不应读取。
- 哪些命令需要人工确认。

不要把真实简历、身份证号、访问令牌或生产日志原样交给模型。

## 七、面试怎么讲 AI Coding

不要只说：

> 我会使用 AI 辅助编程，提高开发效率。

可以说得更具体：

> 我把 AI Coding 用于陌生模块定位、重复代码生成和失败路径测试补充。每次修改都限制范围，审查 diff，并运行项目验证。涉及权限、数据迁移和发布的动作保留人工确认。对于模型生成的代码，我会重点检查接口兼容性、异常路径和敏感信息。

如果有真实数据，再补充：

- 节省多少时间。
- 补充了多少测试。
- 发现过什么错误。
- 哪些任务不适合交给模型。

## 八、自测问题

1. 为什么 AI 生成的代码必须人工审查？
2. 什么任务适合 AI Coding，什么任务需要谨慎？
3. 如何避免模型顺手修改无关文件？
4. 如何保护仓库中的敏感信息？
5. 测试如何帮助你约束 AI 生成代码？
