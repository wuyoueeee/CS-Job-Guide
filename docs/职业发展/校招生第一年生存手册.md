# 校招生第一年生存手册：从学生到工程师

> 写给即将入职的计算机专业应届生。这篇文章不会跟你讲大道理，只讲那些我带过三届校招生之后，发现每个人都应该早点知道的事。

---

## 一、入职前 30 天：做好准备，别浪费黄金窗口

很多同学拿到 Offer 之后就躺平了，想着"反正入职了有人带"。坦白说，入职后的第一个月，mentor 和 leader 对你的预期是很低的——但这恰恰是你建立第一印象的最好窗口。提前准备的人，起跑线真的不一样。

### 1.1 技术栈预习（入职前 2-4 周）

不要泛泛地"学一下微服务"或者"看看 Redis"。你需要非常具体地搞清楚以下问题：

**要搞清楚的事情清单：**

- [ ] 你们组用的技术栈具体是什么版本？（Java 8 还是 17？Spring Boot 2 还是 3？）
- [ ] 核心中间件有哪些？（RPC 框架是 Dubbo 还是 gRPC？消息队列是 Kafka 还是 RocketMQ？）
- [ ] 代码仓库在哪里？用的什么分支策略？
- [ ] 有没有内部框架/脚手架？一般校招生大部分时间是在内部框架上写业务逻辑

**怎么搞到这些信息？** 联系你的 mentor 或者 HR 要一份"新人上手资料清单"，大部分公司都有。如果没有，直接在入职前两周发一封简短邮件给 mentor，问三个问题就好：

> "师兄/师姐你好，我是即将入职的 XXX。想提前了解一下咱们组的技术栈，方便我入职前做些准备。方便的话，能否告诉我：① 咱们组主要用哪些技术栈和中间件？② 有没有推荐的入门资料或者文档可以先看看？③ 入职前我做什么准备最有帮助？"

这三个问题就够了——既显得你有主动性，又不会让人觉得你在过度卷。

**预习的重点排序：**

| 优先级 | 内容 | 花的时间 | 说明 |
|--------|------|----------|------|
| 🔴 最高 | 内部框架/脚手架文档 | 找到就看 | 80% 的业务代码是在内部框架上写的 |
| 🔴 最高 | Git 工作流 + Code Review 流程 | 2-3 天 | 你会每天用到的东西 |
| 🟡 中等 | 核心中间件基础操作（Kafka/RMQ/Redis） | 1 周 | 会用就行，原理后面再补 |
| 🟡 中等 | 公司内部的 CI/CD 和发布流程 | 2-3 天 | 第一次上线不会慌 |
| 🟢 可选 | 业务领域知识 | 有空就看 | 入职后再深入也来得及 |

### 1.2 入职前要准备好的问题清单

准备一个 Notion 文档或者飞书文档，列好你入职后要搞清楚的问题。不要入职之后每天"不知道要做什么"。

**必问清单：**

- [ ] 我们组的业务边界是什么？上下游依赖了哪些团队？
- [ ] 新人上手文档在哪里？有没有推荐的阅读顺序？
- [ ] 测试环境怎么申请？本地怎么跑起来？
- [ ] 代码合并流程是怎样的？谁负责 CR？
- [ ] 发布流程是怎样的？灰度策略是什么？
- [ ] 日志在哪里看？监控大盘在哪里？
- [ ] 组里的 oncall 机制是怎样的？
- [ ] 这一期的 OKR/迭代目标是什么？我负责哪部分？

### 1.3 第一周的实操 Checklist

入职第一天往往在搞权限、装机器、走流程，可能会觉得"啥也没干"。没关系，前三天就是用来"搭环境"的。但你要有一个清晰的 checklist，确保自己走在正确的节奏上。

**Day 1-2：环境 + 权限**

- [ ] 申请所有必要的权限（代码仓库、数据库只读、测试环境、监控系统、日志系统、内部文档）
- [ ] 装好开发环境（IDE、JDK、Maven/Gradle、Docker、数据库客户端）
- [ ] 配置好 Git，搞清楚公司内网的 SSH 密钥配置
- [ ] 加入组里的所有群聊、邮件组、日程订阅
- [ ] 把 mentor 和组里每个人的名字记下来（真的，连名字都记不住会很尴尬）

**Day 3-4：代码 + 文档**

- [ ] Clone 核心仓库，本地编译通过
- [ ] 把项目在本地跑起来，调用一个最简单的接口确认链路是通的
- [ ] 通读新人文档（如果有的话），标记出看不懂的地方，集中问一次
- [ ] 画出你理解的系统调用链路，给 mentor 确认一下
- [ ] 读一遍最近两周的代码提交记录，感受一下大家都在改什么

**Day 5：小任务 + 认识人**

- [ ] 领一个最简单的 bug fix 或者文档修改，走通完整的"开发 → CR → 合并 → 发布"流程
- [ ] 约组里每个人聊 15 分钟（"我刚入职，想了解一下你在做什么，方便我后续找你请教"）
- [ ] 周五前写一篇简短的入职周报，发给 mentor

## 二、试用期的 4 个关键信号

试用期本质上是双向的——公司考察你适不适合，你也要考察公司适不适合。但从"过试用期"的角度来说，你的 mentor 和 leader 其实在看这 4 个信号。

### 信号 1：能不能独立完成一个简单需求

> Leader 心里想的是：**我把一个需求讲清楚之后，能不能放心地不再管它，直到你来跟我说做完了？**

**好的做法：**

- 接到需求后，先用自己的话复述一遍给 mentor 确认理解无误
- 拆成几个小步骤（接口设计 → 写代码 → 单测 → 自测 → 提 MR）
- 把涉及到的文件、数据表、依赖服务都搞清楚再开始写
- 测试 case 覆盖正常、异常、边界三种情况
- 提 MR 前自己 review 一遍 diff，删掉调试用的 `System.out.println`，清理掉多余的 import

**坏的做法：**

- 听完需求就闷头写，写完发现理解偏了
- 不写单测直接提 MR，"功能跑通了就行"
- 代码里残留了一堆 `// TODO` 和注释掉的代码
- 测试只测了正常路径，异常情况（空值、超时、并发）一概不考虑

**核心原则：代码质量 > 速度。** 新手最大的误区是"我写得快就是好"，但实际上一个需要 mentor 花一个小时帮你 review 改 bug 的代码，远比一个多花半天自己打磨好的代码更拖累团队效率。

### 信号 2：遇到问题时怎么求助

> Leader 心里想的是：**这个人遇到困难的时候，是能高效地寻求帮助，还是会变成一个"黑洞需求"？**

这是一个三段式求助法，建议直接照抄：

**第 0 步：先给自己设一个时间上限（如 2 小时）。** 超了就求助，不要死磕。

**第 1 步：自己先尝试。**

- 看文档 / 搜索内部知识库 / 翻代码
- 缩小问题的范围（是配置问题还是代码问题？是网络问题还是逻辑问题？）

**第 2 步：整理信息，带着上下文求助。**

不要这样问：

> "师兄，这个功能跑不通，你能帮我看一下吗？"

而要这样问：

> "师兄，我在做 XX 需求的时候，调用 YY 接口报了 `Connection timeout`。我排查过：① 本地能 ping 通目标服务；② 其他接口调用正常；③ 错误只在本地环境出现，测试环境正常。我猜测可能是本地网络代理的问题，能帮我看一下代理配置吗？"

第 2 步的好处是：你让 mentor 的"启动成本"极低——他一看到你整理的信息，可能 30 秒就能定位问题，而不是从头开始理解"你到底在做什么"。

**第 3 步：带方案提问（进阶要求）。**

> "师兄，这个问题我有两个思路：方案 A 是改配置超时时间，方案 B 是加一层重试机制。我倾向方案 A，因为改动小、风险可控。你觉得呢？"

当你开始带方案提问时，leader 心里会想："这个校招生，有点东西。"

| 提问方式 | Mentor 的感受 | 处理效率 |
|----------|--------------|----------|
| "跑不通了，帮我看一下" | "又来了一个需要我从头理解的问题" | 低——mentor 要花 5-10 分钟先搞清楚上下文 |
| 描述问题 + 给出排查过程 | "这个同学自己试过了，节省了我理解时间" | 中——mentor 可以直接定位 |
| 描述问题 + 排查过程 + 候选方案 | "有思路了，我只需要给个建议" | 高——30 秒到 2 分钟搞定 |

### 信号 3：代码 Review 后怎么改

> Leader 心里想的是：**这个人能不能接受反馈并快速成长？还是会觉得 CR comment 是"找茬"？**

CR 是校招生成长最快的方式，没有之一。你写代码写到 60 分，review 一轮能到 80 分，再 review 一轮能到 90 分。这些 review comments 放在外面是几千块一小时的付费 code review，在公司是免费的——千万别当成批评。

**好的做法：**

- 每条 comment 都回复（哪怕只是 `done` 或者 `已修改`）
- **不理解的 comment 一定追问**——这不是表示你不行，是表示你在认真思考。一个追问比十个 `done` 更有价值
- 如果同一个 comment 被提了两次，记下来，下次写代码时主动检查
- 建立一个自己的"代码检查清单"，把 review 中高频出现的 comment 记下来（比如"判空"、"日志级别"、"事务边界"），每次提 MR 前自己先查一遍
- 利用 CR 的机会反向学习：看到同事的 MR 时也去翻一翻审阅 comments，看看别人怎么写、reviewer 怎么评

**坏的做法：**

- Comment 上有争议时第一反应是争辩/防御："我这样写也没问题吧"
- 改了但不回复 comment，reviewer 不知道你改了没有
- 同样的错误反复出现（这会直接让 mentor 觉得你"不长记性"）
- 把 CR 当形式，随便看看就合并

**建议的 CR comment 回复模板：**

| Comment 类型 | 回复方式 |
|-------------|----------|
| 你认同的改进建议 | "好的，已修改。这里确实用 `Optional` 更安全，之前没考虑到 null 的情况。" |
| 你不理解的建议 | "这里我有点没理解，用 `StringBuilder` 代替 `+` 拼接是出于性能考虑吗？我理解这个场景下只拼接一次，差别不大？方便展开讲一下吗？" |
| 你有不同想法的 | "明白你的意思。我当时的考虑是 XXX，不过你的方案在 YY 场景下确实更合理。这样吧，我先按你的建议改，顺便把 XXX 场景的边界情况也考虑进去。" |

### 信号 4：怎么同步进度

> Leader 心里想的是：**我手上管着 8 个人 6 件事，我不可能每天追在每个人后面问进度。这个人能不能让我"不操心"？**

**好的做法：**

- **主动同步，不等 leader 来问。** 需求有阶段性进展时发一条简短消息在群里或给 leader："XX 需求的接口开发完成，正在写单测，预计明天下午提 MR。目前没有遇到阻塞。"
- **只说关键信息：** 做了什么、做到哪了、有没有风险、预计什么时候完成。四句话就够了
- **有风险提前说，不要等到最后一天。** "这个需求比预期复杂，联调时发现上游接口返回的数据格式不对，需要上游改。我已经跟对方同学约了今天下午对齐，可能会延期半天。"——这样的消息周三发，比周五才说强 100 倍
- 周报不是流水账，后文会展开讲

**坏的做法：**

- 沉默式开发：一周不说一句话，leader 不知道你在干什么
- 好消息自己藏着，坏消息也自己藏着
- 进度报告太啰嗦："我今天上午配置了环境变量，然后发现 JDK 版本不对，然后我重装了一下，然后……"——没有 leader 有时间看这个
- 说"做完了"但没自测、没 CR、没上线——你说的是"代码写完了"，leader 理解的是"需求交付了"

| 场景 | 坏的消息同步 | 好的消息同步 |
|------|-------------|-------------|
| 进度正常 | 不吭声 | "XX 功能开发完成，自测通过，MR 已提：[链接]。可以 review 了。" |
| 遇到阻塞 | 默默自己扛，扛了三天还搞不定 | "XX 功能的联调阻塞了，上游 XX 接口参数对不上，我约了对方下午 3 点对齐。如果搞不定我再找你。" |
| 可能要延期 | 最后一天才说"做不完" | "目前进度 60%，但 XX 部分比预期复杂，可能要延期半天到一天。不会影响本周的发版。" |
| 做完了 | "做完了" | "开发 + 自测完成，MR 已合并，测试环境验证通过。已经通知 QA 测试。预计明天灰度上线。" |

## 三、写代码之外的生存技能

很多校招生觉得"技术好就行了"，但说实话，在大厂里写代码的能力只是基础分。真正让你在组里站稳脚跟的，往往是写代码之外的这些事。

### 3.1 开会：带着笔记本，记录 Action Item

校招生在会议上通常不用发言，但这不代表你可以摸鱼。

**开会时必做的事：**

- [ ] 带上笔记本（纸质的或电脑都行，重点是"记"这个动作）
- [ ] 记录所有你听不太懂的术语、系统名称、人名，会后查或问
- [ ] 如果会上给你分配了任务（Action Item），**当场确认**截止时间和交付标准
- [ ] 会议结束后，把跟你有关系的 Action Item 在群里发一遍确认："确认一下，我本周五前完成 XX 需求的接口文档，对吗？"

**不要做的事：**

- 开会不带东西、不记笔记，纯"听会"
- 会上被问到"你有什么想法"时一脸茫然
- 开完会不知道自己该做什么

### 3.2 写周报：不记流水账

这是新人最容易踩的坑。大部分校招生写出来的周报是这样的（❌ 坏例子）：

> 本周完成了 XX 需求的开发，改了两个 bug，参加了三次会议。

这种周报的问题是：它只说了你做了什么，没说你学到了什么、有什么价值、遇到什么问题。

**周报的正确结构（✅ 好例子）：**

```
【本周完成】
- XX 需求：完成接口开发和单测，MR 已合并。涉及 XX 模块、YY 表，新增 3 个接口。
- Bug 修复：修复了 ZZ 场景下的 NPE 问题，根因是上游未传参数时未做判空处理。

【本周收获】
- 熟悉了 Kafka 消费的幂等处理方案，总结了一篇笔记（附链接）
- 通过 XX 需求的 CR 学到了用策略模式替代 if-else 链的写法

【遇到的问题】
- XX 服务联调时发现文档与实现不一致，已反馈给对应同学，预计下周修复

【下周计划】
- 完成 XX 需求的上线和灰度验证
- 学习 XX 中间件的原理，目标是能回答"它是怎么保证高可用的"
```

**三条铁律：**
1. 不记流水账——"写代码"不是工作内容，"写了什么代码、解决了什么问题"才是
2. 必须有"收获/思考"——哪怕这周只学到了一件事，写出来就是你的积累
3. 提前暴露风险——周报是让 leader 知道"你需要什么帮助"的最佳渠道

### 3.3 1 on 1：提前准备 Agenda

大部分公司都有定期 1 on 1 的机制（跟 mentor 周周聊，跟 leader 月月聊）。千万别把 1 on 1 当成"领导找我谈话"，而是当成"我有 30 分钟让眼前这个资深工程师帮我解决任何问题"。

**1 on 1 前要准备什么：**

- [ ] 回顾一下上次 1 on 1 的结论，这期间有什么变化？
- [ ] 准备 3-5 个具体想讨论的话题（不要太泛，比如"怎么提升技术"就不如"我最近在学 Kafka，但不太理解它的 ISR 机制，能帮我讲讲吗"）
- [ ] 如果对工作内容/方向有困惑，准备好具体场景再来聊
- [ ] 记录这次 1 on 1 的结论和待办事项，下次跟进

**1 on 1 的高质量问题示例：**

| 不要问 | 可以问 |
|--------|--------|
| "我该怎么提升自己？" | "我观察到组里 XX 系统经常在高峰期报警，我想了解一下它的核心瓶颈是什么？有没有可能让我参与优化？" |
| "这个需求怎么做？" | "这个需求我有两个实现思路，A 方案更简单但扩展性差，B 方案引入了设计模式但代码量多一倍。你倾向哪个？为什么？" |
| "我做得怎么样？" | "你觉得我最近 XX 方面做得怎么样？有哪些地方你觉得我可以在下个季度重点提升？" |

### 3.4 线上事故：第一时间同步，不要偷偷修

校招生遇到线上问题的第一反应往往是"糟了，我是不是搞砸了"，然后就想自己偷偷修好，假装什么都没发生。**这是最危险的做法。**

**正确的应急流程（建议截图保存）：**

| 步骤 | 动作 | 说明 |
|------|------|------|
| 1 | 第一时间同步 | 在工作群或者直接告诉 mentor："XX 服务刚发完版之后 XX 接口报 500，我正在看。"不要拖，不要试图自己搞定再说 |
| 2 | 快速止血 | 如果是发布引起的，优先考虑回滚，而不是在线修代码。回滚 > 配置降级 > 热修复 |
| 3 | 求援 | 如果 15 分钟内自己搞不定，立刻拉人帮忙。不要有"搞不定就是能力不行"的心理包袱——让线上问题快速恢复比证明自己厉害重要得多 |
| 4 | 记录时间线 | 几点几分做了什么操作、几点几分发现问题、几点几分恢复——这对后续复盘非常关键 |
| 5 | 写复盘 | 事故复盘不是"检讨书"，而是"如何避免下次再发生"。格式：事件经过 → 根本原因 → 改进措施 → 负责人 + 时间点 |

**一句话总结：线上出问题不可怕，可怕的是你瞒着不说、自己乱修、然后让问题扩大。**

## 四、常见的校招生踩坑（前人的血泪教训）

以下 10 个坑，每一个都来自真实案例。如果你能避开其中一半，你已经比 90% 的同期校招生更优秀了。

| 序号 | 踩坑 | 典型表现 | 为什么危险 | 正确做法 |
|------|------|----------|------------|----------|
| 1 | 把 CR comment 当成批评 | 被提了建议后第一反应是解释和争辩："我这样写也没问题""这是参考了之前的代码" | 让 mentor 和同事觉得你"听不进建议"，之后就不太愿意认真 review 你了——这是校招生最亏的 | 每条 comment 都是免费的资深工程师指导，先接受再思考。不理解就追问，不认同就说明你的考虑并虚心讨论。核心态度是"我在成长"而不是"我是对的" |
| 2 | 卡住了不敢问 | 闷头搞一个问题搞了三天，期间没有发出任何求助信号 | 三天后 mentor 发现你还在原地，不仅浪费了时间，还可能影响项目排期 | 给自己设一个 2 小时的上限，超过就求助。求助时带上你的排查过程和思考（参考前文"三段式求助法"） |
| 3 | 只关心自己的一亩三分地 | 只改自己负责的模块，不知道系统全貌，不了解上下游依赖 | 你永远只能做局部优化，无法做出有影响力的贡献。而且出问题时你根本排不了，因为你不了解链路 | 入职前三个月，给自己安排一个任务：能画出你所在系统的完整架构图（数据流 + 服务依赖 + 核心接口调用链）。多翻翻别人的代码 |
| 4 | 不懂装懂 | 组里在讨论技术方案时，为了"融入"而频频点头，但实际上根本没理解 | mentor 以为你懂了，后续安排任务时按你懂了来安排，结果你做不出来——这才是真正的信任崩塌 | 不会就说不会。你可以说："这里我没太理解，能展开讲一下吗？" 或者 "我对 XX 还不太熟，回去补一下再跟你讨论。" 真诚比假装厉害得多 |
| 5 | 过度承诺 | mentor 问"这个周五能搞定吗"，你虽然觉得够呛但不好意思说，就答应了 | 周五交不出来，mentor 对你失去信任。一次过度承诺的影响远超你的想象——之后你的估时会被自动乘以 1.5 | 诚实估时，留出 buffer。如果你不确定，说："我估计需要 5 个工作日，但 XX 部分有不确定性，可能再加一天。我周二同步一次进度，如果发现搞不定我会提前说。" |
| 6 | 轻视非功能性需求 | 代码能跑通就提 MR，不考虑异常处理、日志、监控、性能 | 你的代码上线后出问题没人能排查，因为你根本没打日志。接口慢了你也不知道，因为你没加监控指标 | 把"写好代码"的标准从"功能跑通"提升到"功能正确 + 日志完整 + 异常处理到位 + 关键路径有监控"。这是从学生到工程师的重要转变 |
| 7 | 不敢拒绝/不敢说"我要学一下" | 接到任务时明明不太会，但不好意思说，硬着头皮接 | 做出来的质量差，反而给人留下"能力不行"的印象 | "这个需求我能做，但 XX 部分我之前接触不多，预计需要花半天时间熟悉一下，整体排期会多一天。"——这是成熟的表现 |
| 8 | 只在群里潜水，从不发言 | 组里的技术讨论、业务讨论从不参与，觉得"自己是新人没资格说话" | 存在感为零。三个月后大家对你的一致印象是"挺好的，就是不太熟" | 从简单的问题开始："这个方案里用 XX 中间件是出于什么考虑？我理解用 YY 是不是也可以？" 问问题本身就是参与 |
| 9 | 把加班等同于努力 | 每天加班到最晚，但白天花大量时间在切换任务、刷手机、看消息上 | 不仅透支身体，还会造成"这个人效率低"的印象 | 先管好注意力。早上到的前 30 分钟不刷消息不看手机，列好今天要做的 3 件事。每完成一件再休息。真正的努力看产出，不看工时 |
| 10 | 不维护自己的工作笔记 | 每次遇到的问题解决了就过了，第二次遇到同样的报错时又得重新 Google | 重复踩坑，成长速度打折 | 从第一天开始维护一个工作笔记（飞书/Notion/语雀都行），记录：遇到的报错及解决方案、常用命令、好用的脚本、CR 中收获的技巧。三个月后你会感谢自己 |

## 五、如何建立信任：靠谱比厉害重要

前三个月，我最担心的不是校招生"能力不够"，而是"不靠谱"。能力可以培养，但信任一旦破裂就很难重建。

### 5.1 靠谱的三个标准

> **靠谱 = 说到做到 + 做不了的提前说 + 小事也认真**

拆开来说：

**① 说到做到**

你承诺周五提 MR，周五就真的提了。你承诺覆盖三种异常情况的测试用例，就真的写了三种。**信任是一口口饭吃出来的，是一次次说到做到积累出来的。** 你做不到 100% 准时没关系（资深工程师也经常估不准），但你每次都提前说。

**② 做不了的提前说**

这才是大多数校招生最需要补的一课。你不好意思拒绝，硬扛了一个你搞不定的需求，最后交不出来——这才是真正的不靠谱。真正靠谱的做法是：在发现搞不定的那一刻就说，哪怕离截止日期还有一整天。

**③ 小事也认真**

文档里少一个标点，日志里写错了方法名，提交的 commit message 写了个 `fix bug`——这些"不重要的小事"在你看来可能没什么，但在 leader 看来，这是"这个人做事习惯怎么样"的信号。

**小事清单（做好了加很多分）：**

- [ ] Commit message 清晰有意义（推荐 [Conventional Commits](https://www.conventionalcommits.org/) 格式：`feat: 新增用户导出功能` `fix: 修复导出时空数据导致的 NPE`）
- [ ] MR 的描述写清楚：改了什么、为什么改、怎么验证的
- [ ] 代码里的注释不说废话（`// 调用 xx 方法` 这种注释没有任何信息量），但关键逻辑、魔法值、临时方案一定要注释
- [ ] 别人请你帮忙 review 代码，24 小时内给反馈，哪怕只是"我今天比较忙，明天帮你看"
- [ ] 群里的消息看到了回一个"收到"，大家对你有"反应速度靠谱"的印象
- [ ] 处理完的事发一条消息闭环："上周说的那个 bug 已经修好上线了，后续一周没再复现"

### 5.2 一个能快速建立信任的策略：认领没人愿意做的"脏活"

组里总有一些活：文档没有人维护、监控报警没人梳理、单元测试覆盖率低、日志格式不统一、工具脚本没人写。这些活不显眼、不出彩、没人抢着做——但这恰恰是校招生建立"靠谱印象"的好机会。

**推荐前三个月认领的"脏活"：**

| 建议认领的脏活 | 能体现什么 | 对团队的长期价值 |
|---------------|-----------|------------------|
| 整理/重写新人文档 | 同理心、文档能力 | 降低后续新人上手成本 |
| 补充单测覆盖 | 严谨性、代码理解力 | 减少回归 bug |
| 梳理监控报警规则 | 系统全局视野 | 减少线上故障响应时间 |
| 写一个自动化脚本 | 工程化思维 | 释放团队重复劳动 |
| 整理接口文档 | 沟通能力、全局理解 | 跨团队协作效率提升 |

**一个真实的场景：**

我带过一个校招生，入职第二周主动提出"组里的新人文档太久没更新了，我能不能边学边更新？" 三周后他交出了一份全新的文档，连老员工都发现了一些之前不知道的坑。这件事直接让他三个月试用期没有任何悬念——因为他在做一件"只有他能做、且对团队长期有用"的事，而且用了心。

这就是——**前三个月不要想着"证明自己厉害"，而是要想着"证明自己有用"。**

## 六、第一年的目标清单

这张清单不是 KPI，而是我自己带校招生时，希望他们第一年结束时能够做到的事。别追求每一项都完美，但每做到一项，你就比三个月前的自己强了一大截。

### 技术能力

- [ ] **能独立完成中等复杂度的需求**：从理解需求 → 拆解任务 → 编码 → 单测 → 自测 → 提 MR → 上线，整个流程不需要 mentor 全程盯着
- [ ] **对所在的系统能画出一张完整的架构图**：包含数据流转、服务依赖、核心接口调用链、关键中间件。画不出来就说明你对系统的理解还不够
- [ ] **有一次独立排查线上问题的经历**：从发现异常 → 定位根因 → 修复上线 → 写复盘报告。经历过一次线上事故的全流程，你的成长抵得上写三个月代码
- [ ] **至少掌握两个"非业务"的深入知识点**：比如 JVM 调优、数据库索引优化、消息队列的原理、分布式锁的实现。这会在你第二年面试和晋升时发挥巨大作用

### 团队影响力

- [ ] **做过一次组内技术分享**（30 分钟即可）：不一定要多高深——"我踩过的 5 个坑"就是很好的分享主题。分享的目的是练习"把一件事讲清楚"的能力
- [ ] **输出的文档/工具被至少一个同事主动使用过**：这说明你真的提供了价值，而不是自嗨
- [ ] **你 review 过别人的代码，并且提出过有价值的建议**：这说明你已经在从"写代码的人"向"看代码的人"转型

### 人际关系

- [ ] **有一位可以随时请教的师兄/师姐**：不是 mentor 那种"有义务带你"的关系，而是你主动建立起来的、对方也愿意帮助你的关系。这个人会在你迷茫时给你最真实的建议
- [ ] **组里每次团建/聚餐你都在**：听起来很蠢，但"合群"在职场里比很多校招生想象的重要。你不需要变成社交达人，但你不能是那个"从来不出现的人"

### 个人习惯

- [ ] **建立了一个持续维护的工作笔记**：哪怕只是飞书文档，每周更新一次。一年后你会知道你积累了多少
- [ ] **每周至少花 2 小时学习"体系化的知识"**：不是刷技术公众号，而是看书、看官方文档、看源码。片段化的信息无法构建深度
- [ ] **养成了一个"写完代码先自己 review 一遍"的习惯**：你提给别人 review 的代码，应该先达到自己能接受的标准
- [ ] **复盘过至少一次失败（需求延期、线上 bug、沟通误解）**：复盘不是写检讨，是把这件事情"为什么会发生"的根因挖出来，然后制定一个改进措施。能做到这件事，你的成长速度会远超同龄人

## 写在最后

我从一个校招生成长过来，也带过好几届校招生。每年看他们在第一年里跌跌撞撞、踩坑成长，最大的感触是：

**那些成长最快的校招生，往往不是最聪明的，而是最"听得进去"、最"沉得下心"、最"不怕丢脸"的。**

- **听得进去**：被提意见时不防御，认真思考然后改进
- **沉得下心**：愿意花时间做小事、写文档、补单测，而不是一心想着做大项目
- **不怕丢脸**：敢在组会上问"这个我不懂，能解释一下吗"，敢在 1 on 1 里说"我最近有点迷茫"

学校和职场的最大区别是：学校里你在跟别人比排名，职场上你只需要比昨天的自己更好。

第一年很长，也很短。把基础打扎实，把习惯养好，把信任建立起来。这些东西会在你的第二年、第三年，持续给你回报。

祝你第一年顺利。如果遇到困难，也别太焦虑——每个人都经历过你现在经历的一切。

> 这篇文章写于 2026 年。如果你觉得有用，欢迎分享给跟你一起入职的同学。
