# AI 应用研发路线图

AI 应用研发的核心不是“会用大模型”，而是能把模型接入真实业务流程，并处理检索、工具、评测、成本、安全和可观测问题。

## 一、能力地图

| 能力 | 要会什么 | 可展示材料 |
| --- | --- | --- |
| Prompt Engineering | 任务拆解、约束、样例、评测 | Prompt 版本和 bad case |
| RAG | 文档解析、切分、召回、重排、引用 | RAG Demo 和评测集 |
| Agent | 工具调用、状态管理、人工确认 | Tool Calling 流程图 |
| 工程化 | API、异步任务、缓存、日志、权限 | 服务架构图 |
| 评测 | 准确率、召回率、人工标注、回归测试 | 评测报告 |
| 安全 | 隐私、越权、Prompt Injection | 风险清单 |

## 二、作品集建议

| 项目 | 核心亮点 | 面试价值 |
| --- | --- | --- |
| 个人知识库 RAG | 检索、引用、评测 | 最容易讲出完整链路 |
| 面试复盘 Agent | 工具调用、结构化输出 | 和求职场景强相关 |
| Prompt 管理平台 | 版本、实验、回滚 | 展示工程化意识 |
| 文档问答系统 | 多格式解析、权限隔离 | 接近企业真实场景 |

参考：[AI 应用研发作品集项目清单](../项目实战/AI应用研发作品集项目清单.md)

## 三、简历表达公式

```text
围绕【业务场景】，设计并实现【AI 应用系统】，
完成【检索 / 工具 / 评测 / 安全】等关键模块，
通过【指标或案例】验证效果，并沉淀【可复用能力】。
```

## 四、面试追问

1. 为什么要用 RAG，而不是直接把材料塞进上下文？
2. 切分策略怎么设计？如何处理表格和代码块？
3. 召回不准时怎么定位？
4. 如何评测回答质量？
5. 如何防止模型编造？
6. 工具调用失败或误调用怎么办？
7. 如何控制 Token 成本和延迟？
