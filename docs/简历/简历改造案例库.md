# 简历改造案例库

这个页面不是提供“万能话术”，而是展示如何把空泛经历改造成可信表达。简历改造的核心是补证据，不是换形容词。

## 案例一：普通 CRUD 项目

| 原始表达 | 问题 |
| --- | --- |
| 使用 Spring Boot、MySQL、Redis 完成后台管理系统 | 只有技术栈，没有业务背景、职责和结果 |

改造思路：

1. 写清系统服务谁。
2. 写清你负责哪个模块。
3. 写清为什么用 Redis / 索引 / 权限设计。
4. 补一个结果或验证方式。

改写版本：

```text
负责后台管理系统的用户权限和操作日志模块，基于 RBAC 模型设计角色、菜单和接口权限；
使用 Redis 缓存用户权限集合，减少重复查询，并通过操作日志记录关键变更，便于后续审计和问题追踪。
```

面试追问：

- RBAC 表怎么设计？
- 权限缓存如何失效？
- 如果用户越权访问，系统如何拦截？

## 案例二：课程设计项目

| 原始表达 | 问题 |
| --- | --- |
| 完成在线商城系统，包含登录、商品、订单等功能 | 功能罗列，像作业说明 |

改写版本：

```text
在在线商城课程项目中负责订单创建和库存扣减流程，设计订单、商品、库存三张核心表；
针对重复提交问题增加订单幂等校验，并用事务保证订单创建与库存扣减的一致性。
```

可补强证据：

- 表结构截图。
- 下单流程图。
- 幂等 key 设计。
- 事务边界说明。

## 案例三：AI Demo 项目

| 原始表达 | 问题 |
| --- | --- |
| 调用大模型 API 实现智能问答系统 | 过于浅，容易被认为只是套壳 |

改写版本：

```text
实现面向校招资料的 RAG 问答 Demo，完成 Markdown 文档解析、分块、向量检索和答案引用；
构建 50 条测试问题用于评估召回质量，并根据 bad case 调整 chunk 大小和 TopK 参数。
```

面试追问：

- 为什么要做文档切分？
- 如何评估回答是否正确？
- 模型编造时怎么处理？

## 案例四：实习杂活

| 原始表达 | 问题 |
| --- | --- |
| 参与接口开发和 bug 修复 | 太泛，无法体现贡献 |

改写版本：

```text
参与订单售后模块接口开发，负责退款状态查询和异常提示优化；
通过补充参数校验和日志字段，缩短测试同学定位问题的时间，并沉淀 6 类常见异常处理方式。
```

改造原则：

- 小任务也要写清上下文。
- 不夸大影响，但要写出判断和动作。
- 能量化就量化，不能量化就写验证方式。
